The spec-to-build pipeline is fundamentally deductive. You start with a known goal, derive a plan, execute the plan. The agent's context is curated before it starts — references, summaries, a build plan with named files and functions. The agent works from knowledge toward code. Uncertainty is low by design. When something goes wrong, you revert and re-plan. The whole system is built to minimize surprise.
The iterate skill is trying to handle something structurally different. You don't know what's wrong until you run it. You can't plan your way through bugs you haven't discovered yet. The agent works from observation toward understanding. Each bug is new information that changes what you know about the system. The context isn't curated up front — it accumulates.
And that accumulation is the problem. In the deductive flow, context is bounded — the spec, the references, the build plan. The agent reads what it needs, builds, done. In the empirical flow, every bug discovered adds context. Every fix changes the system state. Every checkpoint adds a summary of what happened. Over a long enough session, the agent is swimming in accumulated observations, and the older ones are getting stale because the fixes changed the system underneath them.
So the regression risk is doubled. There's direct regression — a fix breaks something else, the classic problem. But there's also context regression — the agent's working model of the system drifts from reality because its context is a mix of current state and historical observations that are no longer true. The deductive pipeline avoids this entirely because it rebuilds context from scratch each time. The empirical flow can't do that without losing the thread of discovery.
Your instinct about one-bug-one-checkpoint makes more sense in this light. It's not just about discipline — it's about context hygiene. Each checkpoint is a chance to crystallize what's true right now and let go of the stale observations that led there. The session report isn't just for resuming later — it's a context reset point. You're essentially doing garbage collection on the agent's understanding.
The question I'd be mulling: is there a natural boundary where an empirical session should become deductive? You discover three bugs, fix them, and now you understand the system well enough to write a spec for what "working" looks like. At that point, do you stop iterating and switch to spec-to-build? The empirical phase would be discovery, and once you've learned enough, you shift to the pipeline that handles known-goal execution better.
